    Microservices Map

    This document provides an overview of all microservices in the DexaMinds platform, their responsibilities, and how they interact with each other.

    Service Architecture

    graph LR
    A[API Gateway] --> B[Auth Service]
    A --> C[User Service]
    A --> D[Content Service]
    A --> E[Search Service]
    A --> F[Notification Service]
    B --> G[(Auth DB)]
    C --> H[(User DB)]
    D --> I[(Content DB)]
    E --> J[(Search Index)]
    F --> K[(Message Queue)]
    L[Analytics Service] --> M[(Data Lake)]
    N[Admin Service] --> B & C & D

    Service Catalog

    Service Language Description Data Store Owner
    API Gateway Go Routes requests, handles auth Redis Platform
    Auth Service Node.js Authentication & authorization PostgreSQL Security
    User Service Java User profiles & management MongoDB Core
    Content Service Python Content management MongoDB Content
    Search Service Java Full-text search Elasticsearch Search
    Notification Service Node.js Email & push notifications Redis Comms
    Analytics Service Python Data processing & analytics BigQuery Data
    Admin Service TypeScript Admin dashboard & tools Various Platform

    Service Dependencies

    Auth Service

    • Depends On: User DB (PostgreSQL)
    • Used By: All services
    • API: REST, gRPC

    User Service

    • Depends On: Auth Service, User DB (MongoDB)
    • Used By: API Gateway, Content Service
    • API: gRPC

    Content Service

    • Depends On: User Service, Content DB (MongoDB)
    • Used By: API Gateway, Search Service
    • API: REST

    API Endpoints

    Auth Service

    • POST /auth/login - User authentication
    • POST /auth/refresh - Refresh access token
    • GET /auth/user - Get current user info

    User Service

    • GET /users/:id - Get user profile
    • PUT /users/:id - Update user profile
    • GET /users/:id/preferences - Get user preferences

    Content Service

    • POST /content - Create content
    • GET /content/:id - Get content
    • PUT /content/:id - Update content

    Data Flow

    1. User Authentication mermaid sequenceDiagram Client->>+API Gateway: POST /auth/login API Gateway->>+Auth Service: Authenticate Auth Service->>User DB: Validate credentials Auth Service-->>API Gateway: JWT Token API Gateway-->>Client: Token Response

    2. Content Creation mermaid sequenceDiagram Client->>+API Gateway: POST /content API Gateway->>+Auth Service: Validate token Auth Service-->>API Gateway: User ID API Gateway->>+Content Service: Create content Content Service->>+User Service: Validate user User Service-->>Content Service: User valid Content Service-->>API Gateway: Content created API Gateway-->>Client: Success response

    Service Discovery

    • Internal: Consul
    • External: Kubernetes Services
    • DNS: .svc.cluster.local

    Development Setup

    Local Development

    # Start all services
docker-compose up -d

# Run tests
make test

# View logs
docker-compose logs -f

    Service Template

    name: my-new-service
language: nodejs
description: Brief description
owner: team-name
dependencies:
  - user-service
databases:
  - postgres
  - redis
endpoints:
  - GET /api/resource
  - POST /api/resource

    Scaling Guidelines

    Stateless Services

    • Use Kubernetes HPA
    • Scale based on CPU/Memory
    • Max 10 replicas per service

    Stateful Services

    • Use StatefulSets
    • Persistent volumes
    • Consider sharding

    Security

    Authentication

    • JWT for service-to-service
    • mTLS for internal communication
    • API keys for external services

    Data Protection

    • Encrypt PII at rest
    • Mask sensitive data in logs
    • Regular security audits
